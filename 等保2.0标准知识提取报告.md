### 1.2 核心概念和框架

**等级保护实施生命周期模型（6阶段循环）**：

```
定级与备案 → 总体安全规划 → 安全设计与实施
    ↑                              ↓
    ↑                     安全运行与维护
    ↑                              ↓
    ←—— 持续改进循环 ——— 应急响应与保障
                                     ↓
                              等级保护对象终止
```

**角色三元组**：
- **运营/使用单位**：系统所有者，全生命周期责任主体
- **信息安全服务机构**：建设方、技术支撑方
- **国家管理部门/主管部门**：监督检查方

**两条主线**：
- 安全技术措施实现线：结构框架 → 安全功能要求 → 性能要求 → 部署方案 → 策略实现
- 安全管理措施实现线：制度建立 → 机构人员 → 过程管理（质量/风险/变更/进度/文档）

### 1.3 与等保基本要求的关联关系

- 实施指南是**过程/方法标准**，回答"怎么做"
- 安全需求分析阶段直接引用 **GB/T 22239（基本要求）** 作为目标指标
- 安全总体设计阶段同时引用 **GB/T 22239** 和 **GB/T 25070（安全设计技术要求）**
- 安全设计与实施阶段的**安全控制开发测试**明确三级以上系统需进行源代码安全审核
- 等级测评阶段引用 **GB/T 28448（测评要求）** 和 **GB/T 28449（测评过程指南）**
- 每个阶段的活动输出（共约50项）构成完整的等保文档体系

### 1.5 实施流程步骤

**总体流程（6阶段 × N子活动）**：

#### 阶段一：定级与备案
1. 行业/领域主管部门制定定级指导意见
2. 运营单位分析等级保护对象重要性
3. 确定定级对象（按业务/资产/区域划分）
4. 按 GB/T 22240 综合评定安全保护等级
5. 组织专家评审、主管部门审批
6. 形成定级报告
7. 向公安机关备案

#### 阶段二：总体安全规划
8. 从 GB/T 22239 提取对应等级的基本安全需求
9. 分析重要资产的特殊保护需求
10. 设计总体安全策略文件
11. 设计安全技术体系结构
12. 设计安全管理体系结构
13. 制定分阶段安全建设目标
14. 规划安全建设项目并估算投资

#### 阶段三：安全设计与实施
15. **技术措施详细设计**：结构框架 → 功能要求 → 性能要求 → 部署方案 → 策略实现
16. **管理措施详细设计**：制度范围 → 行为规范 → 评估完善
17. **采购/开发**：产品采购 or 安全控制自开发
18. **安全控制集成**：方案 → 准备 → 集成 → 培训 → 报告
19. **系统验收**：功能测试 + 性能测试 + 安全性测试
20. **系统交付**：文档提交 + 使用说明 + 服务承诺
21. **管理措施实现**：制度建设 + 机构和人员设置 + 过程管理

#### 阶段四：安全运行与维护
22. 确定运行管理角色（系统管理员/安全管理员/审计员）及职责
23. 建立操作规程并记录操作过程
24. 处理变更（分析 → 审批 → 控制 → 记录）
25. 确定监控对象并收集状态信息
26. 安全状态分析和报告
27. 开展安全自查并制定改进方案
28. 服务商选择、管理和监控
29. 等级测评机构定期测评
30. 配合国家管理部门监督检查

#### 阶段五：应急响应与保障
31. 建立应急组织（指挥/行动/策划/后勤/财务）
32. 安全事件分类分级（参考 GB/Z 20986）
33. 制定应急预案程序
34. 开展应急培训和演练
35. 监测预警（状态信息收集 → 分析 → 报送）
36. 应急响应处置
37. 后期调查评估与改进预防
38. 建立应急保障体系

#### 阶段六：等级保护对象终止
39. 识别需转移/暂存/清除的信息资产
40. 信息资产的安全转移或清除
41. 识别需迁移或废弃的设备，制定处理方案
42. 设备的安全处理并记录
43. 存储介质的清除或销毁
44. 处理方案审批存档

### 2.2 核心概念和框架

**四项测评活动循环模型**（适用于初次和再次测评）：

```
测评准备活动 → 方案编制活动 → 现场测评活动 → 报告编制活动
     ↑                                            |
     └—————— 再次测评时复用上次结果 ————————┘
```

**三级测评角色**：
- **测评机构**：承担具体测评工作的第三方机构
- **测评委托单位**：邀请测评的单位（系统运营/使用单位）
- **测评人员**：实施测评的专业人员

**三大测评原则**：
- **公正性**：不受利益影响
- **科学合理性**：方法科学、过程规范
- **完善性**：覆盖全面、证据充分

**三种测评方法**（详见附录E）：
- **访谈**：与有关人员交流讨论获取证据
- **检查**：文档审查 + 实地察看 + 配置检查
- **测试**：漏洞扫描 + 渗透测试 + 功能/性能测试 + 协议分析

**测评结构双层次**：
- **单项测评**：对每个安全要求项的独立测评（可重复、可再现）
- **整体测评**：从纵深防护和措施互补两个角度综合判断

### 2.3 与等保基本要求的关联关系

- 测评目标和内容依据 **GB/T 22239（基本要求）**
- 测评实施方法依据 **GB/T 28448（测评要求）**
- 测评过程指南本身是 GB/T 28449
- 安全设计技术要求引用 **GB/T 25070**
- 云计算补充引用 **GB/T 31167（云计算安全服务指南）** 和 **GB/T 31168（云计算服务安全能力要求）**
- 工控系统引用 **GB/T 30976（工业控制系统安全）**
- 安全事件分级引用 **GB/Z 20986**
- 测评指标编码规则从基本要求的控制点编码派生

| 数据点 | 值 |

| 检查方法细分 | 3 种（文档审查、实地察看、配置检查） |
| 测试工具类型 | 漏洞扫描 + 渗透测试 + 协议分析 |

**分级测评对象抽样数量**：

| 等级 | 对象覆盖策略 | 同类设备最少抽样 |
|:----:|-------------|:---------------:|
| 一级 | 重点抽查关键设备 | 每类至少 1 台 |

| 四级 | 种类完全覆盖，数量抽样 | 每类至少 3 台 |

**三级渗透测试要求**：基于一般脆弱性的内部和外部渗透攻击
**四级渗透测试要求**：同上，但覆盖更全面

### 2.5 实施流程步骤

#### 活动一：测评准备
1. 测评机构与委托单位签署委托测评协议、保密协议
2. 召开项目启动会
3. 收集定级对象基本信息（组织结构、网络拓扑、资产清单等）
4. 准备测评工具和表单（漏洞扫描器、渗透测试工具、调查表格等）
5. **输出**：《项目计划书》、《调查表格》

#### 活动二：方案编制
6. 对系统构成组件分类（客户端/服务器/网络设备/安全设备/人员/文档）
7. 按"重要性→安全性→共享性→全面性→符合性"五原则确定测评对象
8. 选定测评指标（从 GB/T 22239 对应等级选取）
9. 确定测评内容（每个指标采取何种测评方法）
10. 开发测评指导书（具体的操作指令、命令、判定标准）
11. **输出**：《等级测评方案》

#### 活动三：现场测评
12. 召开现场测评首次会议
13. 实施**访谈**（覆盖所有安全相关人员类型）
14. 实施**文档审查**（检查制度、策略、操作规程是否齐备且一致）
15. 实施**实地察看**（物理环境、人员行为、技术设施）
16. 实施**配置检查**（上机验证配置是否正确）
17. 实施**测试**（漏洞扫描、渗透测试、功能/性能测试、协议分析）
18. 记录测评结果并让被测评方确认
19. 归还测评相关资料
20. **输出**：《测评结果记录》

#### 活动四：报告编制
21. 逐项判定单项测评结果（符合/不符合/部分符合/不适用）
22. 从纵深防护和措施互补角度进行整体测评判定
23. 分析评价被测定级对象存在的风险
24. 形成等级测评结论
25. 编制等级测评报告（含安全隐患、改进建议）
26. 评审测评报告
27. 按分发范围分发报告
28. 过程文档归档保存，清理测评介质上的临时数据
29. **输出**：《等级测评报告》

### 3.2 核心概念和框架

**测评方法三角**（每个测评单元使用一种或多种）：

```
       ┌──────────┐
       │ 访谈     │ 与人员交流取证
       ├──────────┤
       │ 核查     │ 观察、查验、分析
       │  ├─ 文档审查
       │  ├─ 实地察看
       │  └─ 配置检查
       ├──────────┤
       │ 测试     │ 工具产生结果并比对
       │  ├─ 漏洞扫描
       │  ├─ 渗透测试
       │  ├─ 功能/性能测试
       │  └─ 协议分析
       └──────────┘
```

**测评单元结构**（四层嵌套）：

```
测评单元编码（如 L1-PES1-01）
├── 测评指标：引用自 GB/T 22239 的具体要求
├── 测评对象：该指标要检查的具体目标
├── 测评实施：具体的操作步骤（1~N项明细）
└── 单元判定：全部肯定→符合；否则→不符合或部分符合
```

**测评双维度**：
- **测评广度（覆盖面）**：涵盖的控制点数量和类型范围
- **测评深度（强弱度）**：每个点的检查彻底程度

**整体测评两个角度**：
- **纵深防护**：从通信网络→区域边界→计算环境→管理中心的逐层递进
- **措施互补**：技术措施与管理措施的协同效应

### 3.3 与等保基本要求的关联关系

- 每个测评单元的"测评指标"直接引用 **GB/T 22239.1** 中对应要求的原文，标注出处（如 `5.1.1.1`）
- 测评实施完全覆盖 **GB/T 22239.1** 及 **GB/T 25070** 中所有要求项
- 附录C给出了与 GB/T 25070（安全设计技术要求）的具體对应关系
- 编码规则：Lx-AAAy-zz 结构
  - Lx = 第一级~第四级
  - AAA = 安全大类（PES=物理和环境、NCS=网络和通信、DCS=设备和计算、ADS=应用和数据、PMS=管理安全）
  - y = 技术(1)或管理(2)标识
  - zz = 顺序号

### 3.5 实施流程步骤

**测评实施在测评单位层面的操作步骤**：

1. **确定测评对象**：按五原则（重要/安全/共享/全面/符合）选择
2. **确定测评指标**：按等级从 GB/T 22239 中选取对应控制点
3. **开发测评指导书**：将每个测评单元的测评实施编写为可操作指令
4. **实施访谈**：按测评指导书与相关人员交流
5. **实施核查**：文档审查 → 实地察看 → 配置检查
6. **实施测试**：漏洞扫描 → 渗透测试 → 功能/性能测试 → 协议分析
7. **记录证据**：保存截图、日志、测试报告等原始证据
8. **单元判定**：每个测评单元独立判定
9. **整体测评**：从纵深防护和措施互补两个角度综合分析
10. **风险评价**：识别残余风险及其影响
11. **结论形成**：综合全部结果形成等级测评结论
