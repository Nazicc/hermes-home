# ISO27001 信息安全管理体系速览

## 1. ISO27001 概述

### 1.1 核心思想：ISMS-PDCA 循环

ISO/IEC 27001:2013 是信息安全管理体系（ISMS）的国际标准，采用 **PDCA（Plan-Do-Check-Act）** 持续改进模型：

```
       ┌───────────────────────────────────────┐
       │    相关方要求（利益相关方）              │
       └──────────────────┬────────────────────┘
                          ↓
┌─────────────────────────────────────────────────┐
│                    PLAN （策划）                  │
│  · 确定ISMS范围（4.3）                          │
│  · 制定信息安全策略（5.2）                      │
│  · 风险评估与处置计划（6.1）                    │
│  · 设定安全目标（6.2）                          │
└──────────────┬──────────────────┬──────────────┘
               ↓                  │
┌──────────────────────────────┐  │
│  DO （实施）                   │  │
│  · 实施风险处置计划            │  │
│  · 实施控制措施（Annex A）     │  │
│  · 资源与能力管理（7.1/7.2）   │  │
│  · 运行控制（8.1）             │  │
└──────────────┬───────────────┘  │
               ↓                  │
┌──────────────────────────────┐  │
│  CHECK （检查）                │  │
│  · 监视测量分析（9.1）         │  │
│  · 内部审核（9.2）             │  │
│  · 管理评审（9.3）             │  │
└──────────────┬───────────────┘  │
               ↓                  │
┌──────────────────────────────┐  │
│  ACT （改进）                  │──┘
│  · 不符合纠正措施（10.1）      │
│  · 持续改进（10.2）            │
└──────────────────────────────┘
```

### 1.2 与等保2.0的定位差异

| 维度 | 等保2.0 | ISO27001 |
|------|---------|----------|
| **定位** | **合规基线** — 国家强制要求，不做不行 | **持续改进体系** — 自愿认证，持续优化 |
| **驱动方式** | 外部监管驱动（公安、网信办） | 内部管理驱动（管理评审+审核） |
| **关注对象** | 特定信息系统（定级对象） | 组织整体信息安全管理体系 |
| **生命周期** | 定级→备案→建设→测评→整改→复查（线性周期） | 策划→实施→检查→改进（PDCA循环） |
| **核心输出** | 等级测评报告（一次性成果） | SOA + 风险评估报告（持续文件） |
| **覆盖范围** | 10个安全类（技术+管理） | 14个控制域 + PDCA流程 |
| **方法论** | 差距分析（与标准比对） | 风险评估（基于业务影响） |
| **实施路径** | 自上而下（国家标准→行业落地） | 自下而上（组织风险驱动） |

> **一句话总结**：等保是**及格线**（必须达到），ISO27001是**健身计划**（持续变好）。

## 2. SOA 适用性声明

### 2.1 SOA 的定义

**适用性声明（Statement of Applicability, SOA）** 是 ISO27001 认证的核心文件之一，记录了：
- 组织选择的控制目标和控制措施
- 选择/不选择的理由（基于风险评估）
- 当前控制措施的实施状态

### 2.2 SOA 的结构

SOA 基于 Annex A 的 14 个控制域，每个条目记录以下信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| **条款号** | Annex A 子项编号 | A.9.1.1 |
| **控制目标（Objective）** | 该控制项要实现的目标 | 建立资产管理职责 |
| **控制措施（Control）** | 具体控制要求 | 识别资产并维护资产清单 |
| **是否适用** | 适用于不适用 | 适用 / 不适用 |
| **选择依据** | 选择或排除的理由 | 基于风险评估结果 |
| **管理制度** | 对应的管理制度/程序文件 | 《资产管理制度》 |
| **实现方式** | 技术/管理实现细节 | 使用CMDB系统管理资产 |
| **实现状态** | 当前实施情况 | 已实施 / 实施中 / 计划中 |

### 2.3 选择适用的控制项

SOA编制的核心逻辑：

```
风险评估 → 识别风险 → 确定处置方案
                              ↓
接受风险   规避风险   转移风险   降低风险（需选择控制措施）
                                          ↓
                              对照Annex A，选择适用的控制项
                                          ↓
                              记录在SOA中，说明选择理由
```

> Annex A 共包含 **14个控制域、35个控制目标、114个控制措施**。SOA 并非要求所有114项全部适用，而是基于风险评估结果选择。

## 3. ISO27001 的 14 个控制域

### A.5 信息安全策略（2个控制目标）

- 制定、审阅、评估信息安全策略
- 确保策略与业务目标和法律法规一致

### A.6 信息安全组织（2个控制目标）

- 内部组织：角色定义、职责分离、管理层承诺
- 外部各方：安全联络、与监管机构/行业组织联络

### A.7 人力资源安全（3个控制目标）

- 任用前：背景调查、录用条款
- 任用中：安全培训、安全意识教育
- 任用终止/变更：离职交接、访问权限撤销

### A.8 资产管理（3个控制目标）

- 资产清单维护、资产责任人
- 信息分级标识
- 介质处理与安全销毁

### A.9 访问控制（4个控制目标）

- 访问控制策略（按业务和安全要求）
- 用户访问管理（注册/注销/审查）
- 用户责任（密码使用、会话锁定）
- 系统和应用访问控制

### A.10 密码学（2个控制目标）

- 密码策略（加密算法、密钥管理标准）
- 密钥管理（生命周期管理、密钥备份/恢复）

### A.11 物理与环境安全（2个控制目标）

- 物理安全区域（安全周界、门禁、监控）
- 设备安全（位置保护、电源、线缆安全、设备处置与再利用）

### A.12 操作安全（7个控制目标）

- 操作程序和职责
- 恶意软件防范
- 备份
- 日志记录与监控
- 操作软件的管控
- 技术脆弱性管理
- 信息系统审计考虑

### A.13 通信安全（2个控制目标）

- 网络安全管理（网络分段、服务隔离）
- 信息传输安全（加密传输、保密协议）

### A.14 系统获取、开发和维护（3个控制目标）

- 安全需求分析（纳入系统开发需求）
- 开发过程安全（安全编码、环境分离）
- 测试数据保护

### A.15 供方关系（2个控制目标）

- 供方关系中的信息安全（安全协议、第三方评估）
- 供方交付服务管理（服务级别协议SLA、监控）

### A.16 信息安全事故管理（2个控制目标）

- 事故管理职责与程序
- 事故的发现、报告、评估、响应、总结
- 证据的收集和保存

### A.17 业务连续性管理（2个控制目标）

- 信息安全连续性（嵌入BCM体系）
- 冗余（信息处理设施的可用性）

### A.18 符合性（3个控制目标）

- 法律法规符合性（知识产权、数据保护、密码合规）
- 独立评审、技术符合性检查、安全策略与标准的符合性

## 4. ISO27001 vs 等保2.0 映射

### 4.1 等保10个安全类 ↔ ISO27001 14个控制域

| 等保2.0 安全类 | ISO27001 对应控制域 | 映射说明 |
|----------------|---------------------|----------|
| **物理安全** | A.11 物理与环境安全 | 直接对应，涵盖机房、设备、线缆安全 |
| **网络安全** | A.13 通信安全 | 网络分段、边界防护、传输加密 |
| **主机安全** | A.9 访问控制 + A.12 操作安全 | 用户鉴别、日志审计、恶意软件防范 |
| **应用安全** | A.14 系统获取/开发/维护 | 安全需求、安全编码、测试数据 |
| **数据安全** | A.8 资产管理 + A.10 密码学 | 数据分级、加密、介质管理 |
| **安全管理机构** | A.6 信息安全组织 | 组织架构、人员职责、授权流程 |
| **安全管理制度** | A.5 信息安全策略 | 策略制定、审阅、评审 |
| **人员安全** | A.7 人力资源安全 | 聘用/在岗/离职的全过程安全 |
| **系统建设管理** | A.14 + A.15 供方关系 | 开发采购安全、外包管理 |
| **系统运维管理** | A.12 操作安全 + A.16 + A.17 | 运维、事件、灾备连续性 |

### 4.2 核心差异

| 方面 | 等保2.0 | ISO27001 |
|------|---------|----------|
| 管理颗粒度 | 按级别统一要求 | 基于风险评估 |
| 供应商管理 | 弱（仅系统建设管理涉及） | 强（A.15独立控制域） |
| 密码学要求 | 作为数据安全一部分 | 独立控制域（A.10） |
| 业务连续性 | 作为数据备份要求 | 独立控制域（A.17） |
| 合规性 | 不突出 | 独立控制域（A.18） |

## 5. 认证流程

### 5.1 整体路线

```
组织准备 → 体系建立 → 试运行 → 内部审核 → 管理评审 → 认证审核 → 保持证书

    自评阶段                  文审+一审→二审         年审→再认证（3年）
```

### 5.2 详细阶段说明

| 阶段 | 活动 | 关键输出 |
|------|------|---------|
| **1. 体系建立** | 定义ISMS范围、制定安全策略、风险评估、选择控制项 | 风险评估报告、SOA、ISMS手册 |
| **2. 试运行** | 运行控制措施≥3个月，产生运行记录 | 操作日志、事件记录、培训记录 |
| **3. 内部审核** | 组织内部审核员，全面检查ISMS运行 | 内审报告、不符合项清单 |
| **4. 管理评审** | 管理层评审ISMS适宜性、充分性、有效性 | 管理评审报告、改进行动计划 |
| **5. 第一阶段审核（文审）** | 认证机构审查体系文件是否满足ISO27001要求 | 文审报告、问题清单 |
| **6. 第二阶段审核（一审）** | 现场审核：验证体系实际运行与文件的符合性 | 不符合项报告 |
| **7. 纠正验证** | 整改不符合项，关闭所有问题 | 纠正措施验证记录 |
| **8. 颁发证书** | 认证机构颁发ISO27001证书（有效期3年） | 认证证书 |
| **9. 年度监督审核（年审）** | 每年至少1次监督审核，关注变化和持续改进 | 监督审核报告 |
| **10. 再认证** | 3年证书到期前，重新进行认证审核 | 新认证证书 |

### 5.3 认证机构选择

| 考虑因素 | 说明 |
|---------|------|
| **认可范围** | 确认认证机构具有CNAS/UKAS等认可资质 |
| **行业经验** | 优先选择熟悉本行业的认证机构 |
| **审核员资质** | 确认审核员具有IRCA/CCAA注册资质 |
| **审核周期** | 了解从文审到发证的预期时间 |
| **服务口碑** | 参考其他组织的认证体验评价 |

### 5.4 常见问题

```
Q: ISO27001认证需要多长时间？
A: 从体系建立到获得证书通常需要 6-12 个月

Q: 证书有效期多长？
A: 3年，每年需接受监督审核（年审）

Q: 等保通过后能否直接获得ISO27001认证？
A: 不能。等保是合规基线，ISO27001是管理体系认证，
   两者标准体系不同，但等保的基础建设可以支撑ISO27001的部分控制措施
```
