# 金融行业网络安全等级保护实施指引

## 1. 标准概述

### 1.1 背景与定位

JR/T 0071-2020《金融行业网络安全等级保护实施指引》由中国人民银行发布，是在**等保2.0通用要求**（GB/T 22239-2019）基础上的金融行业扩展标准。

**为什么需要金融扩展？**
金融行业核心IT系统（网银、支付、征信、保险、证券等）遭受攻击的后果远超普通行业，涉及：
- 客户资金安全
- 金融系统稳定性（系统性风险）
- 个人敏感金融信息保护
- 监管合规与声誉风险

### 1.2 适用范围

本标准适用于：
- 银行业金融机构
- 证券期货业机构
- 保险业机构
- 第三方支付机构
- 金融科技公司（持牌）
- 其他从事金融业务的机构

### 1.3 与等保2.0的关系

| 维度 | 等保2.0通用要求 | JR/T 0071-2020 金融扩展 |
|------|-----------------|------------------------|
| 安全等级 | 一级至四级 | 保持一级至四级体系 |
| 控制措施 | 通用安全要求 | 通用要求 + 约20%金融专项扩展 |
| 分类方式 | 按系统等级 | F1-F4四维分类（按业务类型） |
| 核心关注 | CIA三性 | CIA + 数据安全 + 业务连续性 + 监管合规 |

## 2. F1-F4 四维分类法

JR/T 0071 创新性地引入四维分类法，根据金融业务类型对系统进行分级定级。

### 2.1 F1：互联网金融服务类

**典型系统：**
- 第三方支付平台
- P2P借贷系统
- 众筹平台
- 互联网理财销售平台
- 互联网保险销售平台

**安全等级建议：** 三级（L3）起步

**重点扩展要求：**
- 强身份认证（至少两种认证因素）
- 交易完整性保护
- 防重放攻击机制
- 互联网侧WAF/抗DDoS

### 2.2 F2：核心业务系统类

**典型系统：**
- 网上银行系统
- 支付清算系统
- 反欺诈系统
- 征信系统
- 证券交易系统
- 保险核心系统

**安全等级建议：** 三级（L3）起步，部分建议四级

**重点扩展要求：**
- 交易数据加密存储
- 同城双活 + 异地灾备
- 实时交易监控
- 统一身份管理（IAM）

### 2.3 F3：重要支撑系统

**典型系统：**
- 数据中心基础设施
- 灾备系统
- 证书管理系统（CA）
- 密码管理系统
- 网络安全审计平台

**安全等级建议：** 三级（L3）起步

**重点扩展要求：**
- 物理环境多重冗余
- 密码设备合规管理（国密算法）
- 认证链路保护
- 运维操作审计

### 2.4 F4：内部管理类

**典型系统：**
- OA办公系统
- 人力资源系统（HR）
- 财务管理系统
- 邮件系统

**安全等级建议：** 二级（L2）起步

**重点扩展要求：**
- 访问控制（最小权限）
- 日志审计
- 终端安全管控

### 2.5 分类对比矩阵

| 维度 | F1 互联网金融服务 | F2 核心业务系统 | F3 重要支撑系统 | F4 内部管理类 |
|------|-------------------|----------------|----------------|--------------|
| 安全等级建议 | L3+ | L3~L4 | L3 | L2 |
| 对外暴露面 | 高（互联网） | 中（专线+网银） | 低（内网） | 低（内网） |
| 数据敏感性 | 高（交易+个人） | 极高（核心账务） | 中（配置+密钥） | 中（员工信息） |
| 连续性要求 | RPO≤15min, RTO≤2h | RPO≈0, RTO≤30min | RPO≤1h, RTO≤4h | RPO≤24h, RTO≤24h |

## 3. 金融扩展要求矩阵

以下是JR/T 0071在通用L3基础上增加的金融行业专项要求（约20%增量项）。

### 3.1 数据安全扩展

| 编号 | 扩展要求项 | 说明 |
|------|-----------|------|
| DS-01 | 数据加密存储 | 核心业务数据必须使用加密存储（AES-256或国密SM4） |
| DS-02 | 透明数据加密（TDE） | 数据库级透明加密，防止存储介质泄露 |
| DS-03 | 令牌化（Tokenization） | 敏感数据（卡号、身份证号）使用令牌替代 |
| DS-04 | 数据脱敏 | 非生产环境使用脱敏数据 |
| DS-05 | 数据分类分级 | 建立金融数据分类分级制度 |
| DS-06 | 跨境数据管控 | 金融数据出境需合规审批 |

### 3.2 业务连续性扩展

| 编号 | 扩展要求项 | 说明 |
|------|-----------|------|
| BC-01 | 同城双活 | 同城数据中心双活架构 |
| BC-02 | 异地灾备 | 异地灾备中心，距离 > 300km |
| BC-03 | RPO/RTO量化 | 明确恢复点目标（RPO）和恢复时间目标（RTO） |
| BC-04 | 容灾演练 | 每年至少2次容灾切换演练 |
| BC-05 | 业务影响分析（BIA） | 定期开展业务影响分析 |

### 3.3 身份管理扩展

| 编号 | 扩展要求项 | 说明 |
|------|-----------|------|
| IM-01 | 统一身份平台（IAM） | 建立企业级统一身份认证平台 |
| IM-02 | 生物识别 | 支持指纹、人脸等生物识别因素 |
| IM-03 | 多因素认证（MFA） | 关键系统强制MFA |
| IM-04 | 特权账号管理（PAM） | 管理员账号纳入PAM管控 |
| IM-05 | 单点登录（SSO） | 内部系统间SSO集成 |

### 3.4 云安全扩展

| 编号 | 扩展要求项 | 说明 |
|------|-----------|------|
| CS-01 | 金融云托管 | 优先使用持牌金融云服务 |
| CS-02 | 租户隔离 | 云租户间强隔离（VPC/专有宿主机） |
| CS-03 | 云安全基线 | 云资源配置需符合安全基线 |
| CS-04 | 云访问安全代理（CASB） | 云应用访问管控 |
| CS-05 | 加密密钥管理（KMS） | 云上密钥托管服务 |

### 3.5 安全运维扩展

| 编号 | 扩展要求项 | 说明 |
|------|-----------|------|
| SO-01 | 安全运营中心（SOC） | 7×24小时安全监控 |
| SO-02 | 威胁情报接入 | 接入金融行业威胁情报共享平台 |
| SO-03 | 渗透测试 | 每年至少1次全面渗透测试 |
| SO-04 | 供应链安全 | 第三方供应商安全评估 |
| SO-05 | 员工安全意识培训 | 每季度至少1次 |

## 4. 金融行业等保重点关注

### 4.1 数据安全

金融行业数据安全是等保合规的**重中之重**，需重点关注：

- **全生命周期保护：** 采集、传输、存储、处理、交换、销毁各环节
- **加密技术栈：** 国密算法（SM2/SM3/SM4）优先于国际算法
- **密钥管理：** 使用硬件安全模块（HSM）保护根密钥
- **数据防泄露（DLP）：** 网络DLP + 终端DLP + 邮件DLP
- **数据库审计：** 独立于数据库服务器的审计系统

### 4.2 业务连续性

金融系统连续性要求远超普通行业：

- **网银系统：** 要求RPO≈0（零数据丢失），RTO≤30分钟
- **支付系统：** 7×24小时可用，容忍停机时间极短
- **同城双活：** 应用层双活，数据库层主备同步
- **异地灾备：** 异步复制，主中心故障时手工/自动切换
- **混沌工程：** 主动注入故障验证系统韧性

### 4.3 身份管理

统一身份管理架构建议：

```
┌─────────────────────────────────────────┐
│              统一身份平台 (IAM)          │
├─────────┬─────────┬─────────┬───────────┤
│ 用户管理 │ 认证管理 │ 授权管理 │ 审计管理  │
├─────────┼─────────┼─────────┼───────────┤
│ 目录同步 │ 密码认证 │ RBAC    │ 登录审计  │
│ 生命周期 │ MFA     │ ABAC    │ 权限审计  │
│ 自助服务 │ SSO     │ 权限隔离 │ 行为审计  │
│          │ 生物识别 │ 最小权限 │          │
└─────────┴─────────┴─────────┴───────────┘
```

### 4.4 云安全

金融上云的安全关注点：

| 关注点 | 说明 |
|--------|------|
| 合规性 | 金融云需通过等保三级及以上认证 |
| 数据主权 | 数据存储位置需符合监管要求 |
| 加密独立性 | 密钥由金融客户自行管理（BYOK） |
| 审计能力 | 云平台需提供完整操作日志 |
| SLA保证 | 云服务SLA需满足监管连续性要求 |

## 5. 典型场景

### 5.1 网银系统等保配置建议

**分类：** F2（核心业务系统类）

**安全等级：** 三级（L3）

**等保配置要点：**

| 安全层面 | 配置项 | 实现方式 |
|----------|--------|---------|
| 物理安全 | 双数据中心 | 同城双活 + 异地灾备 |
| 网络安全 | 分区防护 | 互联网区 → DMZ → 应用区 → 数据区 |
| 网络安全 | WAF部署 | 防护SQL注入、XSS、CC攻击 |
| 主机安全 | 服务器加固 | 最小化安装、主机入侵检测（HIDS） |
| 应用安全 | 强认证 | 口令 + 短信验证码 / 数字证书 |
| 应用安全 | HTTPS | TLS 1.2+，国密SSL |
| 数据安全 | 敏感数据加密 | TDE + 字段级加密 |
| 数据安全 | 日志审计 | 集中日志平台（SIEM） |

### 5.2 支付系统等保配置建议

**分类：** F2（核心业务系统类）

**安全等级：** 三级至四级

**等保配置要点：**

| 安全层面 | 配置项 | 实现方式 |
|----------|--------|---------|
| 身份认证 | 多因素 | 数字证书 + 交易密码 + 生物识别 |
| 交易安全 | 完整性校验 | MAC校验、签名验证 |
| 交易安全 | 防重放 | 时间戳 + nonce + 序列号 |
| 连续保障 | 双活+灾备 | 同城双活（同步复制），异地灾备（异步） |
| 风控安全 | 实时风控 | 交易行为分析、异常检测引擎 |
| 合规需求 | 清结算审计 | 完整的交易流水审计链 |

### 5.3 征信系统等保配置建议

**分类：** F2（核心业务系统类）

**安全等级：** 三级

**重点关注：** 个人征信信息保护（《征信业管理条例》）

| 安全层面 | 配置项 | 实现方式 |
|----------|--------|---------|
| 访问控制 | 最小权限 | 按业务角色授权，无默认全览权限 |
| 数据保护 | 存储加密 | 征信数据全量加密 |
| 数据保护 | 查询审计 | 每笔征信查询记录可追溯 |
| 数据保护 | 数据脱敏 | 展示时部分掩码处理 |
| 接口安全 | API加密 | 征信接口双向认证 + 加密传输 |
| 连续性 | 灾备 | 异地灾备，RTO≤2小时 |
