# PCI DSS 标准速览

## 1. PCI DSS 概述

### 1.1 什么是 PCI DSS

PCI DSS（Payment Card Industry Data Security Standard，支付卡行业数据安全标准）是由**PCI安全标准委员会（PCI SSC）** 制定的全球性数据安全标准，由 American Express、Discover、JCB、MasterCard、Visa 五大卡组织联合成立。

### 1.2 适用范围：CDE

PCI DSS 的保护范围为 **CDE（Cardholder Data Environment，持卡人数据环境）**，包括：

- 处理、存储或传输持卡人数据的人员、流程和技术
- 持卡人数据 = PAN（主账号）+ 持卡人姓名 + 有效期 + 服务代码
- **敏感认证数据**（CVV/CVC、PIN、磁条数据）——严格禁止存储（加密也不允许）

### 1.3 核心原则

PCI DSS 围绕 **6大目标、12项核心要求** 构建：

| 目标 | 要求编号 | 核心要求 |
|------|---------|---------|
| **目标1：构建和维护安全网络** | 要求1 | 部署防火墙 |
| | 要求2 | 不使用厂商默认密码 |
| **目标2：保护持卡人数据** | 要求3 | 保护存储数据 |
| | 要求4 | 加密传输 |
| **目标3：维护漏洞管理程序** | 要求5 | 防恶意软件 |
| | 要求6 | 安全系统和应用 |
| **目标4：实施强访问控制** | 要求7 | 按业务需要知必 |
| | 要求8 | 唯一ID |
| | 要求9 | 物理安全 |
| **目标5：监控和测试网络** | 要求10 | 日志审计 |
| | 要求11 | 定期测试 |
| **目标6：维护信息安全策略** | 要求12 | 策略 |

## 2. 12项核心要求详细说明

### 要求1：部署防火墙

**分项要求：**
- 1.1 建立防火墙配置标准
- 1.2 网络拓扑文档化
- 1.3 CDE 与不受信网络之间禁止直连
- 1.4 互联网入站/出站流量限制

**技术实现：**
- 网络防火墙（NGFW）
- Web应用防火墙（WAF）
- 网络分段（VLAN + ACL）
- DMZ 架构

**常见检查点：**
- [ ] 有全网拓扑图
- [ ] 防火墙规则评审（每季度/每半年）
- [ ] CDE 与公共网络严格隔离
- [ ] 入站规则按"默认拒绝"原则配置

### 要求2：不使用厂商默认密码

**分项要求：**
- 2.1 变更默认密码/参数
- 2.2 建立安全配置标准
- 2.3 非控制台管理使用加密（SSH/TLS）

**技术实现：**
- 安全基线（Security Baseline）
- 配置自动化（Ansible/Puppet/Salt）
- 密码复杂度策略

**常见检查点：**
- [ ] 所有设备默认密码已更改
- [ ] 无线接入点默认SSID已更改
- [ ] 使用加密远程管理（如SSH v2，禁用Telnet）

### 要求3：保护存储数据

**分项要求：**
- 3.1 最小化存储：仅存储必要数据
- 3.2 禁止存储敏感认证数据（CVV/PIN/磁条）
- 3.3 PAN 展示时需掩码
- 3.4 PAN 存储需不可逆（加密/截断/哈希）
- 3.5 加密密钥安全管理

**技术实现：**
- **加密：** AES-256 加密存储
- **截断：** 仅保留前6后4位
- **掩码：** 展示时遮罩中间部分
- **令牌化（Tokenization）：** 用令牌替代 PAN
- **TDE：** 数据库级透明加密

**关键规则（不允许含例外）：**
> **禁止存储**完整磁条数据、CVV/CVC、PIN 及 PIN 块

### 要求4：加密传输

**分项要求：**
- 4.1 公网传输持卡人数据须使用强加密
- 4.2 从不信任证书和未加密的明文传输

**技术实现：**
- TLS 1.2+（禁止 SSL v3 / TLS 1.0）
- SSH（文件传输）
- IPSec VPN
- 内部网络也建议加密（防止内网嗅探）

**常见检查点：**
- [ ] 所有 Web 页面强制 HTTPS
- [ ] 证书有效且未被吊销
- [ ] 无线网络使用 WPA2/WPA3

### 要求5：防恶意软件

**分项要求：**
- 5.1 部署防病毒软件
- 5.2 自动更新签名库
- 5.3 生成审计日志
- 5.4 定期扫描

**技术实现：**
- 端点保护平台（EPP）
- EDR/XDR（端点检测与响应）
- 定期全盘扫描 + 实时防护

**常见检查点：**
- [ ] 所有系统安装 AV/EPP
- [ ] 防病毒日志可审计
- [ ] 补丁管理系统运作中

### 要求6：安全系统和应用

**分项要求：**
- 6.1 及时安装安全补丁
- 6.2 开发安全流程（SDLC）
- 6.3 代码审查
- 6.4 公共可见 Web 应用需防 OWASP Top 10
- 6.5 自定义代码需检查常见漏洞
- 6.6 外部 Web 应用需定期漏扫/代码审计

**技术实现：**
- SAST（静态应用安全测试）
- DAST（动态应用安全测试）
- 代码审查流程
- 补丁管理工具（WSUS/SCCM）

**常见检查点：**
- [ ] 安全开发流程已文档化
- [ ] 补丁在发布后30天内安装（紧急补丁更快）
- [ ] 外部应用每年代码审计

### 要求7：按业务需要知必

**分项要求：**
- 7.1 限制 CDE 访问权限
- 7.2 建立访问授权机制
- 7.3 权限审批流程

**技术实现：**
- RBAC（基于角色的访问控制）
- 最小权限原则
- 权限定期复审

**常见检查点：**
- [ ] 权限审批记录
- [ ] 季度权限复审

### 要求8：唯一ID

**分项要求：**
- 8.1 分配唯一用户ID
- 8.2 适当认证
- 8.3 密码管理
- 8.4 实施双因子认证
- 8.5 共享账号管理（服务账号）

**技术实现：**
- LDAP/AD 统一认证
- 双因子认证（TOTP/SMS/令牌）
- 密码策略：长度 ≥ 7位，字母+数字，每90天轮换
- 失败尝试锁定策略

**常见检查点：**
- [ ] 无共享账号
- [ ] 非控制台管理及远程访问启用双因子
- [ ] 密码历史记录 ≥ 4次

### 要求9：物理安全

**分项要求：**
- 9.1 物理进入控制
- 9.2 访客管理
- 9.3 设备/介质管控
- 9.4 介质销毁
- 9.5 数据备份离线存放

**技术实现：**
- 门禁系统（IC卡/生物识别）
- 视频监控
- 介质销毁（消磁/粉碎）
- 机柜锁

**常见检查点：**
- [ ] 访客登记表
- [ ] 介质销毁记录
- [ ] 数据中心机柜上锁

### 要求10：日志审计

**分项要求：**
- 10.1 审计策略
- 10.2 记录日志内容
- 10.3 日志保护
- 10.4 时间同步（NTP）
- 10.5 日志审查
- 10.6 日志保留（≥1年，在线≥3个月）

**技术实现：**
- SIEM（安全信息和事件管理）
- 集中日志收集（Syslog/Splunk/ELK）
- NTP 时间同步
- 日志完整性保护（WORM）

**常见检查点：**
- [ ] 所有关键系统已接入日志平台
- [ ] 日志保留 ≥ 1年
- [ ] 日志每日审查
- [ ] 时间同步已验证

### 要求11：定期测试

**分项要求：**
- 11.1 无线网络扫描（季度）
- 11.2 内网/外网漏洞扫描（季度 + 变更后）
- 11.3 渗透测试（每年 + 重大变更后）
- 11.4 入侵检测/防御系统（IDS/IPS）
- 11.5 文件完整性监控（FIM）

**技术实现：**
- ASV（Approved Scanning Vendor）季度外部扫描
- 内部漏扫（Nessus/Qualys）
- 渗透测试（第三方红队）
- IDS/IPS（Snort/Suricata）
- FIM（Tripwire/AIDE/Osssec）

**常见检查点：**
- [ ] ASV 季度扫描通过
- [ ] 漏扫高危漏洞修复
- [ ] 渗透测试报告年度更新

### 要求12：信息安全策略

**分项要求：**
- 12.1 信息安全策略文档
- 12.2 风险评估流程
- 12.3 使用政策
- 12.4 安全职责明确
- 12.5 安全培训
- 12.6 供应商管理
- 12.7 事件响应计划
- 12.8 年度安全评估

**技术实现：**
- 安全策略管理系统（GRC平台）
- 员工安全意识培训
- 供应商安全评估问卷
- 事件响应演练（桌面推演 + 实战）

**常见检查点：**
- [ ] 安全策略每年评审更新
- [ ] 所有员工签署安全承诺
- [ ] 供应商合同含安全条款
- [ ] 事件响应计划文档化并演练

## 3. 合规分级

PCI DSS 按**年交易量**划分合规等级：

| 等级 | 交易量标准 | 合规要求 |
|------|-----------|---------|
| **一级（Level 1）** | 年交易量 ≥ 600万笔 | QSA 年度现场评估 + ASV 季度外部扫描 + 渗透测试 |
| **二级（Level 2）** | 年交易量 100万~600万笔 | QSA 年度现场评估（或 SAQ D）+ ASV 季度扫描 |
| **三级（Level 3）** | 年交易量 2万~100万笔 | SAQ（自评问卷）+ ASV 季度扫描 |
| **四级（Level 4）** | 年交易量 < 2万笔 | SAQ（自评问卷）+ ASV 季度扫描 |

**关键角色：**
- **QSA（Qualified Security Assessor）：** 认证安全评估机构，负责年度现场评估
- **ASV（Approved Scanning Vendor）：** 认证扫描服务商，负责季度外部漏洞扫描
- **SAQ（Self-Assessment Questionnaire）：** 自评问卷，适用于低交易量商户

## 4. 与等保2.0的映射关系

虽然 PCI DSS 和等保2.0属不同标准体系，但技术控制点高度重叠：

| PCI DSS 要求 | 等保2.0对应项 | 说明 |
|-------------|-------------|------|
| 要求1（防火墙） | 网络安全-边界防护 | 网络分区、ACL |
| 要求2（默认密码） | 主机安全-配置管理 | 安全基线、最小化配置 |
| 要求3（存储保护） | 数据安全-数据加密 | 加密存储、数据脱敏 |
| 要求4（传输加密） | 通信保密性-加密传输 | TLS/SSH |
| 要求5（防恶意软件） | 主机安全-恶意代码防范 | AV/EDR |
| 要求6（系统安全） | 应用安全-开发安全 + 补丁管理 | SDLC、漏洞管理 |
| 要求7（访问控制） | 访问控制-权限管理 | RBAC、最小权限 |
| 要求8（身份认证） | 身份鉴别-双因子 | 唯一标识、密码策略 |
| 要求9（物理安全） | 物理安全-设备/介质管理 | 门禁、视频监控 |
| 要求10（日志审计） | 安全审计-审计记录 | SIEM、日志保护 |
| 要求11（定期测试） | 测评+渗透测试 | 漏扫、渗透 |
| 要求12（安全策略） | 安全管理制度 | 策略文档、培训 |

**差异点：**

| 维度 | PCI DSS | 等保2.0 |
|------|---------|---------|
| 范围 | CDE（持卡人数据） | 全系统 |
| 分级依据 | 交易量 | 系统重要程度 |
| 评估方式 | QSA年审 / SAQ自评 | 测评机构现场测评 |
| 惩罚 | 罚款、取消收单资格 | 责令整改、行政处罚 |
| 更新频率 | 约3年一个大版本 | 约5年 |

## 5. 持卡人数据保护最佳实践

### 5.1 数据最小化策略

```
仅存储必要数据
├── 允许存储：PAN（加密后）、有效期、持卡人姓名、服务代码
├── 禁止存储：CVV/CVC、PIN、完整磁条数据
└── 展示规范：仅显示前6位+后4位（如 411111******1111）
```

### 5.2 加密方案比较

| 方案 | 安全性 | 性能 | 适用场景 |
|------|--------|------|---------|
| **AES-256 列级加密** | 高 | 中 | 数据库中 PAN 字段 |
| **TDE（透明数据加密）** | 高 | 中低 | 数据库全盘加密 |
| **Tokenization（令牌化）** | 极高 | 高 | 支付处理中的 PAN 替代 |
| **截断（Truncation）** | 中 | 高 | 日志中 PAN 处理 |
| **哈希（Hash）** | 高 | 高 | 不可逆的 PAN 查询匹配 |

### 5.3 令牌化（Tokenization）推荐架构

```
持卡人数据输入
      ↓
┌─────────────────┐
│  Token Vault     │  ← 加密存储 PAN-Token 映射
│  令牌服务         │
└─────────────────┘
      ↓
令牌（Token）→ 业务系统（无需接触明文的PAN）
      ↓
交易处理、对账、退款等
```

**令牌化优势：** 业务系统无需进入 CDE 范围，大幅缩小合规边界。

### 5.4 传输安全配置

| 协议 | 状态 | 最小版本 |
|------|------|---------|
| SSL v3 | ❌ 禁止 | — |
| TLS 1.0 | ❌ 禁止 | — |
| TLS 1.1 | ❌ 禁止（2021年后的版本要求） | — |
| TLS 1.2 | ✅ 接受 | 需使用强加密套件 |
| TLS 1.3 | ✅ 推荐 | 最强安全性 |

## 6. 常见不符合项

### 6.1 高频不符合项 TOP 10

| 排名 | 不符合项 | 涉及的PCI DSS要求 |
|------|---------|------------------|
| 1 | 未定期执行漏洞扫描 | 要求11 |
| 2 | 使用SSL/早期TLS版本 | 要求4 |
| 3 | 存储CVV/CVC（即使加密） | 要求3（严格禁止） |
| 4 | 远程访问未启用双因子认证 | 要求8 |
| 5 | 防火墙规则未定期审核 | 要求1 |
| 6 | 日志未集中管理或保留不足1年 | 要求10 |
| 7 | 补丁管理不及时 | 要求6 |
| 8 | 默认密码未更改 | 要求2 |
| 9 | 缺少安全意识培训 | 要求12 |
| 10 | 持卡人数据存储超过必要范围 | 要求3 |

### 6.2 典型问题案例

**案例1：日志保留不足**
> 某支付机构因日志仅保留3个月（PCI DSS要求≥1年），在QSA年审中被判定为不符合，需紧急扩容日志存储并补全缺失时段。

**案例2：CVV存储隐患**
> 某电商平台为方便退款，在数据库中保留了 CVV 字段。PCI 合规扫描发现后，当天触发数据清理并产生罚款——CVV 在任何情况下都不允许存储。

**案例3：默认密码残留**
> 某收单机构在100+台 POS 终端上未更改出厂默认密码，在 ASV 扫描中被发现，导致季度扫描不通过，需全量整改。
