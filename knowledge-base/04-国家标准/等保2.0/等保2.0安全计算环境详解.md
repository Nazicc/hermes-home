# 等保2.0安全计算环境详解

## 概述

**安全计算环境**是等保2.0技术要求中最核心、控制点最多的安全类。标准将其拆分为两个子类：

1. **设备和计算安全**（面向操作系统、主机、网络设备层）
2. **应用和数据安全**（面向应用系统、数据库、数据层）

在GB/T 22239.1-2019的附录A（规范性附录）中，安全计算环境相关控制点的属性标识如下：

| 控制点 | 所属子类 | S/A/G |
|--------|---------|:----:|
| 身份鉴别 | 设备和计算安全 | **S** |
| 访问控制 | 设备和计算安全 | **S** |
| 安全审计 | 设备和计算安全 | **G** |
| 入侵防范 | 设备和计算安全 | **G** |
| 恶意代码防范 | 设备和计算安全 | **G** |
| 资源控制 | 设备和计算安全 | **A** |
| 身份鉴别（应用） | 应用和数据安全 | **S** |
| 访问控制（应用） | 应用和数据安全 | **S** |
| 安全审计（应用） | 应用和数据安全 | **G** |
| 软件容错 | 应用和数据安全 | **A** |
| 资源控制（应用） | 应用和数据安全 | **A** |
| 数据完整性 | 应用和数据安全 | **S** |
| 数据保密性 | 应用和数据安全 | **S** |
| 数据备份恢复 | 应用和数据安全 | **A** |
| 剩余信息保护 | 应用和数据安全 | **S** |
| 个人信息保护 | 应用和数据安全 | **S** |

**集中管控**（L3新增，属性G）虽独立列出，但在实践中作为安全计算环境的安全管理支撑，本文一并说明。

## 1. 身份鉴别

### 1.1 要求项分级对照

| # | 要求项 | L1 | L2 | L3 | L4 |
|---|--------|:--:|:--:|:--:|:--:|
| a | 用户身份标识和鉴别（唯一性、复杂度、更换周期） | ✅ | ✅ | ✅ | ✅ |
| b | 登录失败处理（结束会话/限制次数/超时退出） | ✅ | ✅ | ✅ | ✅ |
| c | 远程管理鉴别信息传输加密 | — | ✅ | ✅ | ✅ |
| d | **两种或两种以上组合鉴别技术（双因子认证）** | — | — | ✅ | ✅ |
| e | 首次登录强制修改初始口令 | — | — | ✅(应用) | ✅ |
| f | 鉴别信息丢失/失效时的安全重置机制 | — | — | ✅(应用) | ✅ |
| g | 重要操作**再次进行身份鉴别** | — | — | — | ✅ |

### 1.2 关键解析

#### L1 基础要求
- 身份标识具有唯一性
- 口令有基本复杂度要求并定期更换
- 登录失败后采取必要保护措施

#### L2 增强
- 增加远程管理加密要求（防止鉴别信息在网络传输中被窃听）
- 增加登录连接超时自动退出

#### L3 核心新增：双因子认证 ⭐
> **"应采用两种或两种以上组合的鉴别技术对用户进行身份鉴别"**

这是L3监督保护级的标志性要求之一。组合方式包括：
- **所知**（口令/PIN）+ **所有**（USB Key/智能卡/OTP令牌）
- **所知**（口令）+ **所是**（指纹/人脸/虹膜等生物特征）
- **所知**（口令）+ **行为特征**（击键动力学/鼠标轨迹）

**设备层**（设备和计算安全）：操作系统/网络设备登录需双因子
**应用层**（应用和数据安全）：应用系统登录需双因子 + 首次登录强制改密 + 丢失重置机制

#### L4 强化
- 应用层增加 **重要操作再次鉴别**（如转账、权限变更操作需二次认证）

| c | 删除或停用多余过期账号，避免共享账号 | — | ✅ | ✅ | ✅ |
| d | 最小权限原则、权限分离 | — | ✅ | ✅ | ✅ |
| e | 授权主体配置访问控制策略 | — | — | ✅ | ✅ |
| f | 访问控制粒度（用户级/进程级，文件/数据库表级） | — | — | ✅ | ✅ |
| g | **安全标记 + 强制访问控制（MAC）** | — | — | ✅ | ✅ |

### 2.2 DAC 与 MAC 说明

#### 自主访问控制（DAC）— L1/L2基础
- 主体（用户/进程）自主决定客体（文件/资源）的访问权限
- 典型的DAC实现：UNIX文件权限（rwx）、Windows ACL
- **限制**：无法防止木马程序以用户身份复制敏感数据

#### 强制访问控制（MAC）— L3新增 ⭐
> **"应对所有主体、客体设置安全标记，并依据安全标记和强制访问控制规则确定主体对客体的访问"**

| 特性 | DAC（L1/L2） | MAC（L3+） |
|------|:-----------:|:----------:|
| 控制粒度 | 用户/组级 | 安全标记级 |
| 标记体系 | 无 | 等级标签（绝密/机密/秘密/公开） |
| 控制规则 | 用户自主授权 | 系统强制策略（BLP/Biba） |
| 防木马泄密 | 弱 | 强（不可绕过） |
| 实现复杂度 | 低 | 高（需统一标记平台支持） |

**L4强化**：访问控制粒度进一步细化（应用层需达到"记录或字段级"）

### 3.2 关键解析

#### 审计覆盖范围（按等级递增）

| 等级 | 设备层 | 网络层 | 应用层 | 用户行为 |
|:----:|:------:|:------:|:------:|:--------:|
| L1 | 主要设备 | — | — | 登录事件 |

- 审计记录中必须包含 **主体标识** 和 **客体标识**（谁？对什么？做了什么？）
- 审计进程受到严格的保护，防止非授权中断（需用审计管理员独立账号管理）

## 5. 恶意代码防范

### 5.1 分级要求

| 等级 | 要求 |
|:----:|------|
| L1 | 安装防恶意代码软件，及时更新 |
| L2 | + 定期查杀 |
| L3 | + 建立从系统到应用的信任链（可信计算技术） |
| L4 | **采用免受恶意代码攻击的技术措施** 或 **采用可信计算技术建立信任链**，实现重要程序/文件完整性检测，检测到破坏后进行恢复 |

### 5.2 L3/L4 可信计算要求 ⭐

> **"应采用免受恶意代码攻击的技术措施或采用可信计算技术建立从系统到应用的信任链，实现系统运行过程中重要程序或文件完整性检测，并在检测到破坏后进行恢复。"**

**关键技术**：
- **TPM**（Trusted Platform Module，可信平台模块）
- **TCM**（Trusted Cryptography Module，可信密码模块，中国国家标准）
- **信任链**：BIOS → 引导加载程序 → OS内核 → 应用 → 度量、存储、报告

**L4增强**：不仅检测完整性，还要能够自动 **恢复** 被破坏的程序或文件。

|---|--------|:--:|:--:|:--:|:--:|
| a | 校验/加密保证传输完整性 | ✅ | ✅ | ✅ | ✅ |

- 传输完整性：HTTPS/TLS、IPSec、消息认证码（MAC/HMAC）
- 存储完整性：数字摘要（SHA-256）、消息认证码、数字签名
- 通用协议攻击示例：基于HTTP协议的数据篡改、SQL注入

### 7.2 数据保密性

| # | 要求项 | L1 | L2 | L3 | L4 |
|---|--------|:--:|:--:|:--:|:--:|
| a | 加密保证传输保密性 | — | ✅ | ✅ | ✅ |
| b | 加密保证存储保密性 | — | — | ✅ | ✅ |
| c | 专用/安全通信协议防通用协议攻击 | — | — | ✅ | ✅ |
| d | **基于硬件设备进行加解密运算和密钥管理** | — | — | — | ✅ |

**L4独特要求**：
- **"应基于硬件设备对重要通信过程进行加解密运算和密钥管理"**
- 硬件加密（HSM、加密卡、硬件安全模块）相比软件加密具有：
  - 密钥不可导出（防泄露）
  - 加解密运算不占用CPU
  - 更强的侧信道攻击防护

### 7.3 推荐密码技术

| 用途 | 推荐算法（中国商用密码） | 国际等效算法 |
|------|:----------------------:|:-----------:|
| 传输加密 | SM2/SM4 + SM3 | ECDHE + AES-GCM + SHA-256 |
| 存储加密 | SM4（对称加密） | AES-256 |
| 完整性校验 | SM3（密码杂凑） | SHA-256 |
| 数字签名 | SM2（椭圆曲线公钥密码） | ECDSA (P-256) |
| 身份认证 | SM2/SM9（标识密码） | RSA/ECDSA |

## 8. 数据备份恢复

### 8.1 要求项分级对照

| # | 要求项 | L1 | L2 | L3 | L4 |
|---|--------|:--:|:--:|:--:|:--:|
| a | 重要数据**本地备份与恢复** | ✅ | ✅ | ✅ | ✅ |
| b | **异地实时备份** | — | — | ✅ | ✅ |
| c | 重要数据处理系统**热冗余** | — | — | ✅ | ✅ |
| d | **异地灾难备份中心** + 业务实时切换 | — | — | — | ✅ |

### 8.2 备份策略层级

```
L1: 本地备份 ─── 简单数据拷贝
L2: 本地备份 ─── 定期全量+增量备份
L3: 本地备份 ─── 异地实时备份 ─── RPO < 15分钟
     + 热冗余（主备自动切换）
L4: 本地备份 ─── 异地实时备份 ─── 异地灾备中心
     + 热冗余 ─── RPO ≈ 0，RTO < 分钟级
```

> **RPO**（Recovery Point Objective）：恢复点目标，允许丢失的数据量
> **RTO**（Recovery Time Objective）：恢复时间目标，允许中断的时间

### 9.2 技术实现方式

| 场景 | 实现方法 |
|------|---------|
| **内存释放** | 内存清零（memset/ SecureZeroMemory）、GC后显式覆盖 |

| **虚拟化** | 内存页清零、磁盘卷销毁后重建 |

## 11. 个人信息保护

### 11.1 要求项

| # | 要求项 | L1 | L2 | L3 | L4 |
|---|--------|:--:|:--:|:--:|:--:|
| a | **仅采集和保存业务必需的用户信息** | — | — | ✅ | ✅ |
| b | **禁止未授权访问和使用用户信息** | — | — | ✅ | ✅ |

### 11.2 合规要点

- **最小化原则**：只收集业务必需的信息，不采集非必要个人数据
- **访问控制**：用户信息加密存储，仅授权人员可按最小权限访问
- **脱敏处理**：展示时对敏感字段（身份证号、手机号）进行脱敏
- **与《个人信息保护法》关联**：等保2.0的个人信息保护要求与2021年《个人信息保护法》在合规要求上高度一致

## 附录：安全计算环境关键技术术语

| 术语 | 说明 |
|------|------|
| **DAC** | Discretionary Access Control，自主访问控制。由资源所有者决定访问权限 |
| **MAC** | Mandatory Access Control，强制访问控制。系统基于安全标记强制决定访问 |
| **TPM** | Trusted Platform Module，可信平台模块（国际标准） |
| **TCM** | Trusted Cryptography Module，可信密码模块（中国国家标准） |
| **信任链** | 从系统启动到应用运行的完整度量链：BIOS→Boot→OS→App，逐级度量验证 |
| **HSM** | Hardware Security Module，硬件安全模块，用于密钥管理和加密运算 |
| **RPO** | Recovery Point Objective，恢复点目标 |
| **RTO** | Recovery Time Objective，恢复时间目标 |
| **BLP** | Bell-LaPadula模型，经典强制访问控制模型（侧重保密性） |
| **Biba** | Biba模型，经典强制访问控制模型（侧重完整性） |
