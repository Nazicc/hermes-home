# 等保2.0设计技术要求详解（GB/T 25070系列）

## 1. 概述：GB/T 25070 与 GB/T 22239 的关系

### 1.1 标准定位

| 标准 | 编号 | 定位 | 核心内容 |
|------|------|------|----------|
| **基本要求** | GB/T 22239（等保2.0） | "要达到什么" | 各安全等级的**安全控制要求**（安全计算环境、安全区域边界、安全通信网络、安全管理中心） |
| **设计技术要求** | GB/T 25070（等保2.0） | "具体怎么做" | 实现上述安全控制要求的**技术设计方案**（架构设计、机制设计、策略设计） |

GB/T 25070 是对 GB/T 22239 的**落地指导**。基本要求规定了"要达到什么安全目标"，设计技术要求规定了"用什么样的技术方案来实现这些目标"。

### 1.2 标准构成

GB/T 25070 系列被划分为多个部分：

| 部分 | 内容 | 覆盖领域 |
|------|------|----------|
| 第1部分 | 通用设计要求 | 通用信息系统，替代 GB/T 25070-2010 |
| 第2部分 | 云计算安全要求 | 云计算平台安全设计 |
| 第3部分 | 移动互联安全要求 | 移动互联安全设计（本知识文档未覆盖） |
| 第4部分 | 物联网安全要求 | 物联网系统安全设计 |
| 第5部分 | 工业控制系统安全要求 | 工业控制系统安全设计 |

### 1.3 核心设计框架

等保2.0安全技术设计框架以 **"一个中心、三重防护"** 为核心：

```
┌─────────────────────────────────────────────────────┐
│                  安全管理中心                         │
│          （系统管理 · 安全管理 · 审计管理）           │
└─────────────────────────────────────────────────────┘
         ▲                    ▲                    ▲
         │                    │                    │
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ 安全计算环境  │◄──┤ 安全区域边界  ├──►│ 安全通信网络  │
│ (节点/应用)   │   │ (边界防护)   │   │ (传输安全)   │
└──────────────┘   └──────────────┘   └──────────────┘
```

- **安全计算环境**：对信息进行存储、处理及实施安全策略的部件
- **安全区域边界**：安全计算环境边界上实施安全策略的部件
- **安全通信网络**：安全计算环境之间进行信息传输及实施安全策略的部件
- **安全管理中心**：对上述三部分的安全机制实施**统一管理**的平台（第二级及以上需要）

### 1.4 设计过程要求

等保安全技术设计应遵循以下过程：

1. **梳理业务流程** → 了解系统现状、特点及特殊安全需求
2. **风险评估** → 识别资产、威胁和脆弱性，结合等级保护标准提出防护需求
3. **梳理主、客体及其权限** → 明确主体对客体的最小访问权限
4. **分层分域设计** → 区域划分，明确计算环境、区域边界、通信网络、安全管理中心的位置
5. **分析关键保护点** → 从操作系统、数据库、应用系统、网络等层面分析
6. **安全机制及策略设计** → 在关键保护点上进行安全机制及策略设计

## 2. 通用设计框架

### 2.1 多级安全域划分

GB/T 25070 第1部分定义了从**第一级到第五级**逐步增强的安全保护环境设计。每一级在低一级的基础上**增加或增强**安全功能。

各级设计目标与策略：

| 等级 | 设计目标 | 关键增强特性 |
|------|----------|--------------|
| **第一级(L1)** | 自主访问控制，用户对所属客体的自我保护 | 用户身份鉴别、自主访问控制(用户组级)、常规完整性校验、恶意代码防范 |
| **第二级(L2)** | 更强的自主安全保护能力 | 增加**系统安全审计**、**客体安全重用**、用户数据**保密性保护**、**安全管理中心** |
| **第三级(L3)** | 统一安全策略管控下保护敏感资源 | 强制访问控制(标记)、**双因素鉴别**、可信计算信任链、网络可信连接、配置可信检查 |
| **第四级(L4)** | 抗渗透能力，结构化安全保护 | **形式化安全策略模型**、结构化部件设计(关键/非关键)、关键参数明确定义 |
| **第五级(L5)** | 更强的抗渗透能力 | 访问监控器（抗篡改、可分析可测试），设计技术要求**另行制定** |

### 2.2 安全计算环境设计

各级安全计算环境设计技术要求汇总：

| 安全功能 | L1 | L2 | L3 | L4 |
|----------|:--:|:--:|:--:|:--:|
| 用户身份鉴别 | 口令鉴别 | 受控口令/其他机制 | **安全管理中心控制**、双因素组合 | **双因素+不可替代鉴别数据** |
| 自主访问控制 | 用户/用户组级 | 用户级 | 用户级，客体粒度到记录/字段 | 同L3，扩展至所有主体 |
| 标记和强制访问控制 | — | — | **安全管理员标记**、强制访问控制 | **扩展至所有主客体** |
| 系统安全审计 | — | 查询/分类/存储 | 分析/报警/防破坏 | 可终止违例进程 |
| 数据完整性保护 | 常规校验 | 常规校验 | **密码技术**支持，可恢复 | 同L3 |
| 数据保密性保护 | — | 密码技术支持 | 同L2 | 同L3 |
| 客体安全重用 | — | ✓ | ✓ | ✓ |
| 恶意代码防范 | ✓ | ✓ | ✓ | ✓ |
| 程序可信执行保护 | — | — | 信任链(可信计算) | **必须构建**信任链 |
| 网络可信连接保护 | — | — | 平台身份鉴别+完整性校验 | 同L3 |
| 配置可信检查 | — | — | 基准库+实时监控 | 同L3 |

### 2.3 安全区域边界设计

安全区域边界逐级增强：

- **L1**：包过滤 + 边界恶意代码防范
- **L2**：增加**边界安全审计**、**边界恶意代码网关**、**边界完整性保护**(探测非法外联)
- **L3**：增加**区域边界访问控制**(自主+强制)、边界审计**集中管理+报警**
- **L4**：边界访问控制**扩展到所有主客体**，审计可**做出相应处置**，结构化设计

### 2.4 安全通信网络设计

安全通信网络逐级增强：

- **L1**：数据传输完整性保护(常规校验)
- **L2**：增加**通信网络安全审计**、数据传输**保密性保护**
- **L3**：完整性保护**可恢复**、增加**通信网络可信接入保护**
- **L4**：审计可**做出相应处置**、可信接入**必须采用**

### 2.5 信任链设计

L3 及以上引入信任链机制：

- **程序可信执行保护**（L3可选，L4必须）：构建从操作系统到上层应用的**信任链**，实现可执行程序的完整性检验，防范恶意代码攻击
- **网络可信连接保护**（L3起）：设备连接网络时进行平台身份鉴别、平台完整性校验、数据传输的保密性和完整性保护
- **配置可信检查**（L3起）：将系统安全配置信息形成**基准库**，实时监控或定期检查配置修改行为

## 3. 安全域模型

### 3.1 分级安全域设计方法

安全域设计遵循 **"分层分域"** 原则，基于业务重要性、数据敏感度、访问控制需求进行划分。

**通用设计方法**：

1. 将系统根据业务功能划分为不同的**安全计算区域**
2. 每区域具有明确的安全边界
3. 区域间通过安全区域边界实施访问控制
4. 同一区域内的系统共享相同的安全策略

### 3.2 常见安全域类型

| 安全域类型 | 说明 | 安全等级建议 |
|------------|------|-------------|
| **核心区** | 存储最敏感数据、核心业务系统 | L3/L4 |
| **生产区** | 日常业务处理系统 | L2/L3 |
| **外联区** | 与外部系统互联的区域 | L2/L3 |
| **DMZ区** | 对外提供服务的区域（Web服务器等） | L2 |
| **管理区** | 系统管理、运维操作区域 | L3 |
| **测试区** | 开发测试环境 | L1/L2 |

### 3.3 不同等级下的安全域建议

| 等级 | 安全域划分建议 |
|------|----------------|
| **L1** | 无需严格划分，全系统作为一个安全计算环境，包过滤即可 |
| **L2** | 建议划分2-3个区域（如核心区、外联区），区域边界实施包过滤+审计 |
| **L3** | 必须严格区域划分，各区域实施**自主+强制访问控制**，统一安全管理中心管控 |
| **L4** | 严格结构化划分，区分关键保护部件和非关键保护部件，形式化安全策略模型 |

### 3.4 定级系统互联设计

相同或不同等级的定级系统之间的互联必须通过：

1. **安全互联部件**：通过通信网络交换网关连接各定级系统
2. **跨定级系统安全管理中心**：统一管理跨系统的安全策略、系统管理、审计管理

设计目标：确保用户身份真实性、操作安全性和抗抵赖性，按安全策略严格控制信息流向。

## 4. 四类扩展设计

### 4.1 云计算安全设计（第2部分）

#### 4.1.1 设计架构

云计算安全设计基于**四层架构**：

| 层次 | 说明 | 安全责任主体 |
|------|------|-------------|
| **硬件设施层** | 计算、存储、网络设备及运行环境 | 云服务商 |
| **资源层(抽象层)** | 网络资源、计算资源、存储资源池，资源管理与调度 | 云服务商 |
| **服务层** | 面向用户提供 IaaS/PaaS/SaaS 服务 | 云服务商+租户 |
| **用户层/访问层** | 云服务使用者访问界面 | 租户 |

云计算平台安全保护等级**不应低于**其承载的最高级别云租户业务系统等级。

#### 4.1.2 虚拟网络设计与租户隔离

**VPC（虚拟私有网络）设计**：

- 支持云租户**私有网络**，不同租户间实现**安全隔离**
- 租户可按业务需求划分安全区域，设置区域间访问控制规则
- 保证虚拟机只能接收到目的地址包括自己地址的报文
- 当虚拟机迁移时，**访问控制策略随其迁移**
- 实现租户级别的服务质量保障机制，避免网络资源过量占用

**管理平面/业务平面分离**：

- 构建**管理网络**，连接管理工具和被管设备
- 保证云计算平台**管理流量与云租户业务流量分离**
- 禁止通过互联网直接访问云计算平台物理网络

#### 4.1.3 云计算安全等级设计差异

| 功能领域 | 第二级(L2) | 第三级(L3) | 第四级(L4) |
|----------|------------|------------|------------|
| 身份鉴别 | 受控口令/简单机制 | **双因素组合鉴别**，鉴别信息保密性+完整性 | 同L3增强 |
| 数据保护 | 完整性保护 | **增加保密性保护**、数据备份与恢复、**虚拟机迁移数据保护** | 同L3 |
| 虚拟化安全 | 基本隔离 | CPU/内存/存储严格隔离、**虚拟机逃逸检测**、镜像快照保护 | 同L3 |
| 安全管理中心 | 基本审计 | 集中监控、**集中审计**、安全策略统一管理 | 动态安全能力扩展 |
| 入侵防范 | — | 暴力破解检测、**Web漏洞检测**、虚拟机逃逸告警 | 同L3 |
| 审计管理 | 基本审计 | **租户审计**、第三方审计接口、云服务创建/删除操作审计 | 同L3 |

#### 4.1.4 云计算安全区域划分

云计算平台安全区域可分为：

- **云服务商管理区域**：资源池整体隔离防护，仅授权人员访问
- **租户资源隔离区域**：保证租户资源隔离，防止跨租户非授权访问
- **外部可访问区域(安全区域A)**：允许外部直接访问
- **内部子网区域(安全区域B)**：不允许外部直接访问
- **不同租户间隔离区域**：服务层+应用层协同部署

### 4.2 物联网安全设计（第4部分）

#### 4.2.1 设计架构

物联网系统采用 **"感知层 → 网络层 → 应用层"** 三层架构，安全设计重点在感知层。

物联网安全设计增加以下专用内容：

- **感知设备和感知层网关的身份标识与鉴别**
- **感知层网络数据传输完整性、保密性、新鲜性保护**
- **异构网安全接入保护**
- **区域边界准入控制**
- **区域边界协议过滤与控制**

#### 4.2.2 感知层安全设计

**身份鉴别**：

| 等级 | 鉴别要求 |
|------|----------|
| L1 | 基本设备标识，常规校验 |
| L2 | 安全管理中心统一管理设备标识、数据来源鉴别**(密码技术支持)**、防止身份伪造和克隆 |
| L3 | **双向身份鉴别**(感知层网关↔感知设备)、**组认证**(认证代理减少拥塞)、数据防虚假注入 |
| L4 | 同L3 + **专用通信协议或安全通信协议服务** |

**轻量级密码**：

物联网感知设备资源受限，必须使用**轻量级密码**：

| 密码类别 | 推荐算法类型 | 用途 |
|----------|-------------|------|
| 轻量级数字摘要算法 | 国家密码行政主管部门规定 | 感知层数据传输完整性保护 |
| 轻量级签名算法 | 国家密码行政主管部门规定 | 数据来源鉴别 |
| 轻量级对称加密算法 | 国家密码行政主管部门规定 | 感知层数据传输保密性保护 |
| 轻量级非对称加密算法 | 国家密码行政主管部门规定 | 感知层数据传输保密性保护 |

**传感数据分级**：

- **生存信息**：感知设备自身状态（电量、在线状态、位置等）
- **鉴别信息**：设备身份标识和鉴别数据
- **隐私性数据**：涉及个人隐私的采集数据
- **重要业务数据**：业务关键数据和指令

不同等级的安全要求不同。L2要求常规完整性校验，L3要求密码技术支持的完整性校验+可恢复，L4增加专用协议保护。

#### 4.2.3 物联网安全等级设计差异

| 功能领域 | L1 | L2 | L3 | L4 |
|----------|:--:|:--:|:--:|:--:|
| 设备身份标识 | — | ✓统一管理 | ✓统一+双向鉴别 | ✓同L3 |
| 组认证 | — | — | ✓认证代理 | ✓同L3 |
| 数据新鲜性 | ✓(序列号/时间戳) | ✓ | ✓ | ✓ |
| 异构网接入 | 基本接入认证 | +入侵检测 | +密码技术保护 | 同L3 |
| 区域边界恶意代码防范 | — | 防恶意代码网关 | 同L2 | +网络扫描/DoS/中间人攻击检测 |
| 区域边界协议过滤 | — | 协议合规检查 | **内容过滤**+白名单 | 深度检测+命令级控制 |
| 数据可用性保护 | — | — | — | ✓关键设备和线路冗余 |

### 4.3 工业控制安全设计（第5部分）

#### 4.3.1 工控网络分层设计（Purdue模型）

工业控制系统采用 **五层架构（Purdue模型）**：

```
企业资源层 (第4层)  ── ERP、MES（本标准不涉及）
     ▲
生产管理层 (第3层)  ── 计划排产、先进控制、仓储管理
     ▲
过程监控层 (第2层)  ── 操作员站、工程师站、历史数据库、实时数据库
     ▲
现场控制层 (第1层)  ── PLC、DCS控制器、RTU、IED
     ▲
现场设备层 (第0层)  ── 传感器、执行器、HMI
```

等保第5部分覆盖 **第0层至第3层**（第4层为通用信息系统不在本部分范围）。

#### 4.3.2 安全区域和管道

**分区原则**：

根据业务系统或其功能模块的**实时性、使用者、主要功能、设备使用场所、业务关联性、广域网通信方式、对工业控制系统影响程度**等划分安全区域。

**推荐安全区域划分示例**：

```
┌──────────────────────────────────────────┐
│          安全管理中心                      │
│  (系统管理 · 安全管理 · 审计管理)         │
└──────────────────────────────────────────┘
         ▲                ▲
         │                │
┌────────────────┐  ┌────────────────┐
│  监控层安全区域   │  │  控制层安全区域   │
│ (工程师站/操作站) │  │ (控制器/PLC等)   │
│      L2/L3      │  │      L3/L4      │
└────────────────┘  └────────────────┘
         │                │
         ▼                ▼
┌──────────────────────────────────────────┐
│               安全通信管道                  │
│    (加密隧道/协议白名单/深度包检测)        │
└──────────────────────────────────────────┘
```

**安全区域和管道特点**：

- **第0-2层可组成一个安全区域**，或**第0-1层组成一个安全区域**
- 同层中可划分为不同安全区域
- 区域间通过安全通信管道实现受控数据交换
- 管道技术包括：IPSec VPN、协议白名单、深度包检测
- 在主要安全区可根据操作功能进一步划分**子区**

#### 4.3.3 控制层安全设计

**控制过程完整性保护**（工控特有）：

设计者必须理解并保护控制回路处理机制：

- 确保在规定时间内完成规定任务
- 防止数据被非法篡改、丢失、延误
- 保护同步机制、校时机制、控制周期稳定性
- 保护现场总线轮询周期稳定性

**需识别的攻击行为**：

- 超过正常数量的中断请求（一个控制周期内）
- 超过合理包速率范围的 ICMP 协议请求
- 超过合理包速率范围的广播报文
- 破坏现场总线**请求-应答机制**
- 破坏冗余工作机制
- 干扰表决器等安全保护系统
- 恶意触发冗余系统切换、安全保护系统停机

**故障隔离措施**：当控制系统遭攻击无法保持正常运行时，将危害控制在最小范围，使系统导向**预先定义的安全状态**。

**现场设备安全设计要点**：

- 角色基础访问控制：设备收到操作命令后检验用户角色权限
- 轻量级密码：适应**实时性高、报文短、计算能力有限**的特点（如流加密算法）
- 现场设备数据包完整性校验：加入**序列号、时间戳**
- 控制设备（PLC/控制器）唯一性标识管理，防止未经授权修改
- 通信网络异常监测：流量、工作周期、抖动值、运行模式、冗余状态

#### 4.3.4 工控安全等级设计差异

| 安全功能 | L1 | L2 | L3 | L4 |
|----------|:--:|:--:|:--:|:--:|
| 设备身份鉴别 | — | 主机设备+工控设备标识 | **双向鉴别**+加密传输保护 | 同L3 |
| 控制过程完整性 | — | ✓识别攻击+故障隔离 | 同L2增强 | **形式化** |
| 访问控制 | 自主访问控制 | **角色基础访问控制(**现场设备) | 同L2 | 强制+结构化 |
| 安全审计 | — | 现场设备审计+远程传输 | 增强审计+分析 | 关联分析+预测预警 |
| 恶意代码防范 | ✓ | 防恶意代码网关 | 同L2 | 深度检测与分析 |
| 区域边界防护 | 包过滤 | 会话状态检测+完整性验证 | 安全通信管道 | 结构化安全互联 |
| 安全管理中心 | — | ✓系统管理+审计管理 | **三权分立**(系统/安全/审计) | +态势感知与预警 |
| 无线攻击防护 | — | — | 电磁屏蔽/检测/防护 | 同L3 |
| 配置变更管理 | — | — | 安全设备+控制器配置管理 | 同L3 |

#### 4.3.5 Purdue模型中的安全域布局

工控系统推荐安全设计分层分区结构结合了：

1. **四层分层**（Purdue模型：现场设备层→现场控制层→过程监控层→生产管理层）
2. **安全区域划分**（横向上按业务重要性和功能划分不同安全区域）
3. **等级保护防护类别**（安全计算环境、区域边界、通信网络、安全管理中心）

架构特点：

- **纵向分层**：每层具有不同的实时性要求和安全策略
- **横向分区**：相同层中不同功能模块可划分为不同安全区域
- **三重防御**：计算环境安全 + 区域边界防护 + 通信网络安全
- **多级互联**：安全管理中心统一管控

## 5. 与等级的关系：不同安全等级的设计策略差异

### 5.1 设计复杂度对比

| 维度 | L1 | L2 | L3 | L4 |
|------|:--:|:--:|:--:|:--:|
| 安全功能数量 | 最少(3项) | 中等(7项) | 最多(10项+) | 与L3相当+结构化 |
| 密码技术强度 | 无要求 | 推荐使用 | **必须使用** | 同L3+专用协议 |
| 访问控制模型 | 自主访问控制 | 自主访问控制 | **强制访问控制** | **形式化强制访问控制** |
| 身份鉴别因素 | 单因素(口令) | 单因素(受控) | 双因素组合 | 双因素+不可替代 |
| 安全管理中心 | — | ✓(基本) | ✓(三权分立) | ✓(结构化) |
| 信任链 | — | — | ✓(可选) | ✓(必须) |
| 系统结构化 | — | — | — | ✓(关键/非关键部件) |

### 5.2 各等级适用场景

| 等级 | 典型场景 | 业务影响 | 关键设计特征 |
|------|----------|----------|-------------|
| **L1** | 个人办公系统、非关键小型业务 | 损害较小 | 自主访问控制、口令保护 |
| **L2** | 一般企业业务系统、中小型应用 | 损害中等 | 审计、客体重用、安全管理中心 |
| **L3** | 政务系统、金融系统、大型企业核心 | 损害严重 | 强制访问控制、双因素、可信计算 |
| **L4** | 国家级关键信息基础设施 | 损害特别严重 | 形式化模型、结构化设计、抗渗透 |

### 5.3 等级与安全区域设计

不同等级对安全区域设计的影响：

| 等级 | 区域划分 | 边界防护 | 安全策略 |
|------|----------|----------|----------|
| **L1** | 不强制划分 | 包过滤即可 | 简单策略 |
| **L2** | 建议划分 | 包过滤+审计+完整性保护 | 自主策略 |
| **L3** | **必须严格划分** | 强制访问控制+信任链 | 统一安全策略(安全管理中心管控) |
| **L4** | 结构化划分 | 强制访问控制+结构化设计 | 形式化安全策略模型 |

### 5.4 等级与信任链

| 等级 | 程序可信执行 | 网络可信连接 | 配置可信检查 |
|------|:------------:|:------------:|:------------:|
| L1 | — | — | — |
| L2 | — | — | — |
| L3 | 可选(推荐可信计算) | 可用(可选) | ✓(基准库+监控) |
| L4 | **必须**(信任链) | **必须采用** | ✓(同L3) |

## 附录：关键概念定义

| 术语 | 定义 |
|------|------|
| **定级系统** | 已确定安全保护等级的信息系统（第一级至第四级） |
| **安全计算环境** | 对定级系统信息进行存储、处理及实施安全策略的部件 |
| **安全区域边界** | 在安全计算环境边界上实施安全策略的部件 |
| **安全通信网络** | 安全计算环境之间信息传输及安全策略实施的部件 |
| **安全管理中心** | 对安全策略及安全机制实施**统一管理**的平台（L2及以上需要） |
| **跨定级系统安全管理中心** | 对相同/不同等级定级系统间互联安全策略实施统一管理 |
| **定级系统互联** | 通过安全互联部件和跨定级系统安全管理中心实现的安全连接 |
| **信任链** | 从操作系统到上层应用的完整性检验链条（L3+引入） |
| **强制访问控制(MAC)** | 基于安全标记的访问控制（L3+引入） |
| **自主访问控制(DAC)** | 用户对其创建的客体具有权限（所有等级） |
