# 网页防篡改系统

## 产品概述

天融信网页防篡改系统是面向政府门户、新闻媒体、在线教育、医疗挂号等 Web 站点推出的网页内容安全防护产品，采用"核心内嵌 + 文件级驱动 + 事件触发"三重防护技术，实时监控 Web 站点目录的增、删、改操作，在毫秒级发现篡改行为并自动恢复（阻断+恢复联动）。产品支持静态页面（HTML/JS/CSS）与动态页面（JSP/ASP/PHP/ASPX）的全面防护——动态页面通过输出内容签名验证，静态页面通过文件内核驱动级监控。

作为等保2.0 合规中"网页防篡改"要求的核心落地产品，该系统与 TopWAF 联动形成"应用层+文件层"双重防御：WAF 在外围防御 Web 攻击，防篡改在核心保护文件完整性。

## 核心功能

| 功能模块 | 说明 |
|---------|------|
| **内核级文件监控** | 基于操作系统内核的文件系统过滤驱动（Windows Filter Driver/Linux LSM），对受保护目录的写操作实时拦截 |
| **自动恢复机制** | 发现篡改后自动从未被篡改的备份目录拉取正确版本恢复，恢复时间 ≤ 300ms |
| **发布与保护分离** | 支持"发布模式"（允许管理员正常更新内容）与"保护模式"（禁止任何非法修改）切换 |
| **动态页面保护** | 基于输出内容 Hash 签名的动态页面完整性校验，确保动态生成页面未被篡改 |
| **多种恢复策略** | 本地备份恢复 / FTP 远程备份恢复 / 共享存储备份恢复 |
| **篡改告警** | 篡改事件及时告警（邮件/短信/Syslog/SNMP Trap），含篡改文件路径、时间、操作类型等信息 |
| **发布管理** | Web 发布系统（WCM/CMS）联动，管理员通过授权通道完成内容更新，非授权修改一律拦截 |
| **文件完整性校验** | 支持对受保护文件的定期完整性校验（SHA256），检测隐蔽篡改（服务器 rootkit 隐藏的篡改） |

## 适用场景

- **政府门户网站**：省/市/县级政府门户（信息公开/在线办事/政民互动）防止页面被篡改为非法信息。
- **新闻媒体网站**：防止新闻首页/重要报道页面被篡改发布虚假新闻。
- **在线招考系统**：高考/中考/考研/公务员考试查分页面、录取结果页面防篡改（每年都有篡改攻击事件）。
- **医疗挂号/疫苗预约**：防止挂号价格、号源信息、预约名单被篡改。
- **教育/高校官网**：防止录取信息/成绩查询页面被篡改。

## 实战Tips（安服工程师经验）

> **云南区域经验：**
> - 云南省政府网站群场景：各厅局网站内容通过省级统一 CMS 发布，但 CMS 发布到 Web 服务器的通道需要防篡改系统开放"授权发布"窗口。建议在防篡改系统上配置 "IP 白名单（CMS 服务器 IP） + 定时发布窗口"——仅在每天 9:00-11:00 和 14:00-17:00 允许修改，其余时间全员保护。
> - 云南高校（即将到来的招生季）成绩查询页面防篡改：云南大学/昆明理工/云南师范等高校的查分页面每年 7-8 月是高危期，建议将查分页面目录和查分接口 URL 列为"重点保护对象"，输出内容签名校验频率从默认 5 秒调整为 1 秒。
> - 云南旅游政务网站（云南省文旅厅/各州市文旅局）注意——网站被篡改后挂载非法旅游广告（虚假旅行社/低价游）是常见威胁。防篡改系统建议配合 TopWAF 联动：WAF 发现异常写入请求立即通知防篡改强化该 URL 的监控力度。

> **贵州区域经验：**
> - 贵州省政府网站集中部署在"云上贵州"平台，Web 服务器群使用共享 NAS 存储静态文件。在此架构下，防篡改系统的备份目录不得放置在同一个共享存储上——恢复时会覆盖原文件。建议备份目录存放在本地 SSD 或独立的备份存储上。
> - 贵州茅台酒销售预约/抽签系统防篡改场景：预约结果页面、中签名单页面是篡改高危点。建议防篡改+WAF 双重保护+人工巡检（每天 3 次人工核对关键页面内容）。
> - 贵州大数据博览会（数博会）官网在每年 5 月举办前是攻击高峰期，建议在 4 月中旬至 6 月中旬启动"最高防护策略"——所有保护目录的文件扩展名（.html/.php/.jsp/.asp）全覆盖，恢复策略设为"自动恢复+紧急告警（电话通知）"。

> **通用经验：**
> - 发版更新是网页防篡改系统最容易"误伤"业务的环节——开发/运维同事发版时忘了先关闭防篡改保护，导致文件修改被拦截，发布失败。建议建立标准的"发版 SOP"：发版前 → 查询防篡改状态 → 暂停保护 → 发布内容 → 验证 → 开启保护。
> - 动态页面保护使用输出内容 Hash 签名方案时，注意动态页面中如果有"时间戳"或"随机数"类动态元素，每次输出的 Hash 都会不同导致误告警。解决方案：在模板中将动态元素剥离出去，仅对页面的固定内容部分做签名。
> - 不要以为只有 ".html" 文件才需要保护——JS/CSS/图片被篡改同样会造成严重危害（挂马/钓鱼/信誉受损）。建议将整个 Web 站点目录（含所有子目录）加入保护范围，仅排除 CMS 发布目录和临时文件目录。
