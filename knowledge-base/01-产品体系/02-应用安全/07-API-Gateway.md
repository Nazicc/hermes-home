# API 安全网关

## 产品概述

天融信 API 安全网关是面向 API 经济时代打造的 API 安全防护与管理产品，专注于 RESTful、GraphQL、gRPC 及 WebSocket 等主流 API 协议的全生命周期安全管控。产品提供 API 资产自动发现、API 请求/响应验证、身份认证与授权、访问速率控制、敏感数据脱敏、异常行为检测等核心能力，解决微服务架构下 API 数量爆炸带来的"影子 API "、"僵尸 API "、数据泄露等安全问题。

API 安全网关采用"前置网关 + 旁路分析"混合架构：在线网关处理南北向 API 流量（认证、限流、校验），旁路分析引擎基于 API 访问日志进行行为建模与威胁检测。产品可与 Spring Cloud/Kubernetes/Service Mesh 等微服务基础设施无缝集成，是企业的API安全防线——在数据泄露发生前的最后一道关卡。

## 核心功能

| 功能模块 | 说明 |
|---------|------|
| **API 资产自动发现** | 自动扫描与发现全网 API 端点，识别影子 API、僵尸 API、过期 API 并标记风险 |
| **请求/响应验证** | 基于 OpenAPI 3.0/JSON Schema 的请求参数校验与响应体安全检查 |
| **身份认证与授权** | OAuth 2.0 / JWT / API Key / mTLS / SAML 等多种认证协议，支持自定义 Token 校验 |
| **速率限制与配额管理** | 基于 API / 应用 / 用户的请求频率控制与调用配额管理 |
| **敏感数据检测与脱敏** | 识别响应中的身份证号、手机号、银行卡号、密钥等敏感数据，支持动态脱敏 |
| **API 异常行为检测** | 基于机器学习的 API 调用行为基线建模，自动发现 DDoS API、参数遍历、数据爬取等异常 |
| **API 版本管理** | 支持多版本 API 共存与灰度发布，兼容性检查与版本废弃管理 |
| **日志审计与溯源** | 全量 API 调用日志记录（请求/响应/耗时/状态码），支持与 SIEM/态势感知对接 |
| **API 安全报告** | 自动生成 API 安全风险报告，包含 API 资产清单、风险等级、攻击事件统计等 |

## 适用场景

- **微服务架构 API 安全**：微服务之间的 API 调用（东西向流量）与对外暴露的 API（南北向流量）统一安全管控。
- **开放平台/开发者门户**：对外提供 API 给第三方开发者时，实现认证/授权/限流/配额管理与审计。
- **移动端 App 后端 API 防护**：移动 App 与后端 API 通信中的防逆向、防重放、防参数篡改。
- **政务数据开放平台**：政府数据开放 API（如天气/交通/社保查询）的安全发布与调用审计。
- **金融 Open API 平台**：满足金融行业开放银行/开放 API 安全技术要求（如 JR/T 0197-2020）。

## 实战Tips（安服工程师经验）

> **云南区域经验：**
> - 云南省级政务数据共享交换平台（云南政务数据开放平台）有数百个数据 API 对省市县三级开放，建议开启"配额管理"功能——省厅级配额 10,000 次/天，州市级 5,000 次/天，县区级 1,000 次/天，并设置超配额自动告警而非直接拒绝（给业务留有余地）。
> - 云南旅游行业（景区/酒店/旅行社）与 OTA 对接场景中，API 网关需仔细核对 OTA 平台的回调 URL 和 IP 白名单，防止 API 被非授权第三方调用导致价格/库存信息泄露。
> - 云南高校信息化建设中（如云南大学、云南民族大学），统一身份认证 API 接口极易被暴力枚举用户名，建议 API 网关在登录 API 上启用"失败次数阈值锁定"——同一 IP 连续 5 次失败锁定 15 分钟。

> **贵州区域经验：**
> - 贵州大数据交易所的 API 商品交易平台，API 网关需支持"按次计费"与"按订阅计费"两种配额模式，API 网关的配额统计精度必须达到"分钟级"以支撑实时计费需求。
> - 贵州省政务数据资源目录（云上贵州平台）的场景，建议开启 API 网关的"敏感数据脱敏"功能——返回给第三方的涉及个人隐私的数据自动脱敏（如身份证号中间 8 位替换为 *），仅在授权应用白名单内返回原始数据。
> - 贵州中小银行（贵州银行、贵阳银行、农信社）的开放银行 API 场景，建议启用 mTLS 双向认证+JWT 双因素认证，满足银保监会《商业银行互联网贷款管理暂行办法》的 API 安全要求。

> **通用经验：**
> - API 网关上线前，必须完成"所有 API 端点的 OpenAPI 规范文档"编制与校验。缺少规格定义的 API 无法进行有效的参数校验与安全检测，形同虚设。建议将 OpenAPI 规范纳入 CI/CD 门禁，API 上线前必须通过规格合规检查。
> - 切勿开启"宽大模式"——当 API 网关检测到请求不符合规范时，总是想要在不影响业务的前提下尽可能放行。这反而掩盖了 API 的安全问题。建议初期设为"告警+阻断混合模式"：高风险违规直接阻断，中低风险告警并记录，定期复盘优化。
> - 针对移动端 API 的重放攻击防范：在 API 请求中引入 Timestamp + Nonce 机制并配置 API 网关的时间窗校验（推荐 5 分钟内有效），可有效防止攻击者截获 API 请求后重放。
