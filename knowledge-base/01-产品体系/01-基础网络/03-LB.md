# TopSec ADC 应用交付控制器（负载均衡）

## 产品概述

TopSec ADC（Application Delivery Controller）是天融信推出的应用交付与负载均衡产品，集服务器负载均衡（SLB）、全局负载均衡（GSLB）、链路负载均衡（LLB）、SSL 卸载、HTTP 加速、应用健康检查与 Web 应用防火墙等能力于一体。产品采用专用 ASIC+FPGA 加速架构，支持 L4/L7 七层协议深度解析，在实现业务高可用的同时提供应用层的安全防护。

TopSec ADC 突破了传统负载均衡仅做流量分发的局限，提供智能流量调度、应用性能优化、安全加固三位一体的应用交付解决方案，广泛服务于政府、金融、运营商、教育等行业的在线业务系统。

## 核心功能

| 功能模块 | 说明 |
|---------|------|
| **四层负载均衡** | 基于 VIP+端口的 TCP/UDP 流量分发，支持轮询/最小连接/最快响应/源地址哈希/IP 哈希等算法 |
| **七层负载均衡** | 基于 HTTP/HTTPS 协议的 URL、Cookie、Header、HTTP Method 等内容交换 |
| **SSL 卸载** | 终结前端 HTTPS，将后端流量降为 HTTP，释放后端服务器加密解密算力开销 |
| **TCP 优化** | TCP 多路复用、连接池、缓冲优化、慢启动优化，提升弱网环境传输效率 |
| **健康检查** | 支持 ICMP/TCP/HTTP/HTTPS 端口探测 + 自定义脚本检查，判定故障节点并自动摘除/恢复 |
| **会话保持** | Source IP、Cookie（插入/被动/重写）、SSL Session ID、Universal（自定义字段） |
| **GSLB** | 基于 DNS 的多数据中心全局流量调度，支持地理位置/就近性/权重策略 |
| **WAF 轻量防护** | 内置基础 WAF 引擎，防御 SQL 注入、XSS、CSRF 等常见 Web 攻击 |
| **链路负载均衡** | 多 ISP（电信/联通/移动/教育网）出口智能选路，出站流量按策略调度，入站流量通过智能 DNS 发布 |

## 适用场景

- **Web 业务高可用**：政府门户/企业官网/在线教育/电商平台的 Web 服务器负载均衡与故障自动切换。
- **数据库读写分离**：MySQL/PostgreSQL 数据库的读库集群负载均衡与健康检查。
- **多数据中心容灾**：主备/双活数据中心架构下的全局流量调度与自动故障转移。
- **多 ISP 链路优化**：企业或数据中心多运营商出口链路的入站/出站智能选路。
- **SSL 集中卸载**：拥有大量 HTTPS 业务（数百个域名）的场景，统一管理 SSL 证书并卸载加解密负担。

## 实战Tips（安服工程师经验）

> **云南区域经验：**
> - 云南省级政府网站群（如云南省政府门户、各厅局网站），通常部署在省政府统一数据中心，建议采用"双机 HA + GSLB 多数据中心"架构——昆明主中心 + 大理灾备中心，保证地震等极端情况下业务不中断。
> - 在线旅游服务平台（携程云南、飞猪云南）的大促期间流量波动剧烈，建议七层负载均衡开启请求限流（Rate Limiting）与连接数限制，防止瞬时峰值压垮后端。
> - 云南教育行业（云南大学、昆明理工等高校）选课季并发极高，建议会话保持采用 Cookie 插入模式而非 Source IP 模式（高校 NAT 出口单一，Source IP 会导致选课请求全部集中到同一台后端服务器）。

> **贵州区域经验：**
> - 贵安新区数据中心集群（华为云/腾讯云/苹果 iCloud 数据中心周边配套）的链路负载场景，建议启用 BGP + ADC 出站智能选路，根据成本/带宽利用率/时延多维度自动选择最优出站链路。
> - 贵州政务云（云上贵州）承载厅局业务系统数百个，ADC 作为统一流量入口，建议配置虚拟服务（Virtual Server）按业务重要性分优先级，核心业务（如社保/公积金）获得更多连接和带宽资源。
> - 贵州中小型银行/农信社的双活数据中心场景，GSLB 的 DNS TTL 建议设置为 60-120 秒，兼顾故障切换速度和 DNS 缓存效率，避免 TTL 过长导致故障切换后用户仍访问已下线站点。

> **通用经验：**
> - ADC 上线前务必完成真实服务器的健康检查验证——建议分步启用：先监听模式观察，确认健康检查结果正确后再启用调度。常见的坑是健康检查配置了错误端口或 URL，导致所有后端被判定故障从而无业务可用。
> - SSL 卸载场景下，后端服务器日志记录的是 ADC 内网 IP 而非真实客户端 IP，需要在 ADC 上配置 X-Forwarded-For （XFF）插入，并在后端 Web 服务器日志格式中增加 %{X-Forwarded-For}i。
> - 会话保持配置不当是"登录后反复被踢回登录页"的最常见原因。检查应用侧的会话机制（Cookie vs Session），确保负载均衡的会话保持策略与应用一致。
