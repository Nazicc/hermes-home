# 数据脱敏系统

## 产品概述

天融信数据脱敏系统（Data Masking System）是面向**生产数据安全使用**的专业数据脱敏产品。系统通过静态脱敏、动态脱敏、数据水印三大核心技术，在**数据使用环节**对敏感信息进行变形、遮蔽、替换或加密处理，确保敏感数据在开发测试、数据分析、数据共享等场景中不被泄露。

产品支持对结构化数据（数据库表、Excel/CSV文件）和非结构化数据（文档、图片中的敏感信息）的脱敏，内置丰富的脱敏算法（遮盖、替换、泛化、扰乱、加密、删除等），保证脱敏后数据的可用性和一致性。满足数据安全法、个人信息保护法对数据脱敏的合规要求。

## 核心功能

| 功能模块 | 说明 |
|---------|------|
| 静态脱敏 | 将生产数据库的全量或部分数据脱敏后，生成脱敏数据集用于开发测试、培训等场景 |
| 动态脱敏 | 数据库访问实时脱敏，根据用户角色自动执行脱敏策略，不影响源库数据 |
| 数据水印 | 在脱敏数据中嵌入不可见水印（数字指纹），数据泄露后可溯源到责任人 |
| 算法库 | 内置100+脱敏算法：遮盖（身份证号中间位*）、替换、随机、泛化（年龄->年龄段）、扰乱、加密 |
| 敏感数据发现 | 自动扫描数据库表结构和数据内容，识别敏感列（姓名、手机号、身份证等） |
| 一致性脱敏 | 同一原始值在脱敏后保持一致（如用户张三->用户张三', 全库替换保持一致） |
| 脱敏策略编排 | 可视化脱敏流程设计，支持多表关联、多阶段脱敏、脱敏前后数据一致性校验 |
| 数据源适配 | 支持Oracle/MySQL/SQL Server/PostgreSQL/DB2/达梦/人大金仓及主流大数据平台 |
| 脱敏调度 | 支持定时脱敏任务、手动脱敏、API触发脱敏等灵活调度方式 |

## 技术规格

| 参数项 | 规格 |
|-------|------|
| 脱敏吞吐 | 500 ~ 50,000 行/秒（取决于脱敏算法和硬件配置） |
| 脱敏算法 | 100+ 内置算法，支持自定义算法开发 |
| 数据源类型 | 结构化（关系型数据库）+ 非结构化（文件、文档、图片OCR） |
| 并发脱敏任务 | 10 ~ 100 并发任务 |
| 部署方式 | 旁路代理（动态脱敏）/ 离线脱敏（静态脱敏） |
| 动态脱敏延迟 | <= 5ms（毫秒级延迟） |
| API接口 | Restful API，支持脱敏策略管理和脱敏任务触发 |
| 硬件支持 | x86 / 国产化服务器 |

## 适用场景

- **开发测试数据脱敏**：从生产库抽取数据脱敏后供开发、测试、培训团队使用
- **数据共享/交换脱敏**：政府部门间、企业间数据共享时的敏感数据脱敏处理
- **大数据分析脱敏**：数据仓库/数据湖中敏感数据分析前的自动脱敏
- **外包/第三方数据服务**：向第三方提供数据服务时的脱敏处理，防止数据泄露
- **BI报表脱敏**：领导驾驶舱、BI分析报表中的敏感数据动态脱敏展示
- **运维查询脱敏**：运维人员数据库查询时，非授权敏感数据自动脱敏显示

## 实战Tips（安服工程师经验）

**云贵区域特色：**

1. **贵州政府数据共享脱敏**：贵州是国家政务数据共享试点省份，政府部门间数据共享场景频繁。建议对共享出去的身份证号、手机号、家庭住址等个人信息采用**动态脱敏**方式，根据不同部门的授权级别设置不同的脱敏策略（如公安部门看到完整身份证号，统计局只能看到年龄段）。

2. **云南旅游大数据分析脱敏**：云南省文旅厅拥有大量游客数据用于旅游大数据分析。建议在数据入湖之前执行**静态脱敏**，生成脱敏后的游客行为数据集供分析使用，原始数据加密存储。

3. **医院脱敏场景**：昆明医科大学第一附属医院、贵州省人民医院等三甲医院在临床研究数据使用时，需要对患者病历（PHI）进行脱敏处理。脱敏时需注意**一致性脱敏**——同一患者在不同表/不同时间的数据，脱敏后要保持一致，才能做关联分析。

4. **数据水印溯源**：贵州大数据交易所的数据交易场景，建议对交易数据集嵌入**可见+不可见双重水印**，当购买方数据泄露时，可通过水印提取信息精准追溯到购买方身份和交易时间。

5. **脱敏性能关注**：贵州大数据企业数据量动辄数百TB，静态脱敏需考虑执行时长。建议使用**分片并行脱敏**功能，对大表按主键分片后多线程并行处理，可大幅缩短脱敏窗口。

6. **脱敏验证**：脱敏完成后务必运行**脱敏有效性验证**——使用脱敏系统内置的识别引擎扫描脱敏后数据，确认敏感数据已被正确脱敏，并验证主外键关系、数据完整性约束是否被破坏。

7. **开发测试环境数据管理**：云南金融、运营商客户的开发测试环境往往管理松散，建议建立**"脱敏后数据生命周期管理"**制度——脱敏数据在开发测试环境使用后，定期清理回收，避免脱敏数据长期散落在测试环境中。
